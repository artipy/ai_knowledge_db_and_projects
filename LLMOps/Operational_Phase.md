# Операционная фаза LLMOps (Operational Phase)

Операционная фаза — это этап жизненного цикла LLM-приложения, на котором готовая система выводится в продакшн, поддерживается и улучшается непрерывно.

Общий контекст жизненного цикла: [[Introduction_to_LLMOps_and_Ideation_Phase]]
Предыдущий этап: [[Development_Phase]]

---

## Внедрение (Deployment)

Внедрение — это процесс предоставления приложения всем пользователям, то есть вывод в продакшн.

Главный фактор при выборе стратегии внедрения — **инфраструктура**. Приложение включает множество компонентов, каждый из которых должен быть интегрирован в единую систему.

### Пошаговый план внедрения

| Шаг | Задача |
|-----|--------|
| **1. Платформа** | Выбор платформы: облако (cloud), self-hosting или on-premise — в зависимости от требований компании |
| **2. Компоненты и API** | Определение, какие компоненты системы должны быть доступны для коммуникации. Каждая часть системы может иметь свой собственный API endpoint, что улучшает масштабируемость |
| **3. Способ запуска** | Выбор способа запуска компонентов: контейнеры, бессерверные сервисы или облако. Каждое решение имеет свои плюсы и минусы: цена, расширяемость, гибкость, эффективность |

## CI/CD

CI/CD автоматизирует интеграцию, тестирование и внедрение.

### Continuous Integration (CI)
1. Получение кода
2. Сборка Docker-контейнера со всем кодом
3. Тестирование контейнера
4. Регистрация контейнера в registry

Весь процесс запускается автоматически при каждом изменении кода.

### Continuous Delivery (CD)
1. Извлечение контейнера из registry
2. Тестирование (smoke tests, integration tests)
3. Внедрение в **staging** (промежуточная среда)
4. Внедрение в **production**

### Масштабирование

После внедрения может возникнуть необходимость в масштабировании при росте нагрузки. Особенно критично для self-hosted решений, использующих GPU.

| Тип масштабирования | Суть | Аналогия |
|---------------------|------|----------|
| **Горизонтальное** | Увеличение количества серверов или контейнеров | Добавить больше такси на дороги |
| **Вертикальное** | Увеличение мощности одного сервера | Улучшить комфорт и скорость одного такси |

## Мониторинг и наблюдаемость (Monitoring & Observability)

После внедрения необходимо непрерывно следить за тем, что система работает корректно.

- **Мониторинг** — непрерывное наблюдение за работой системы
- **Наблюдаемость (Observability)** — предоставление данных из внутренних систем наружу, для информирования администраторов о состоянии системы

Вместе они позволяют проактивно выявлять и решать проблемы.

### Инструменты наблюдаемости

| Инструмент | Описание |
|------------|----------|
| **Логи (Logs)** | Хронология событий, происходящих в системе |
| **Метрики (Metrics)** | Цифровое представление работы системы |
| **Трейсы (Traces)** | Путь запроса через всю систему — от пользователя и обратно |

### Категории мониторинга

**Входной мониторинг (Input Monitoring)**
Отслеживание изменений, ошибок или несоответствующего контента на входе. Ключевой аспект — **смещение данных (data drift)**, требующее постоянного обновления системы.

**Функциональный мониторинг (Functional Monitoring)**
Отслеживание состояния всей системы в целом: здоровья, производительности, стабильности. Включает время ответа, downtime и error rates. Важно учитывать, что цепи и агенты могут быть непредсказуемы и генерировать множество вызовов.

**Выходной мониторинг (Output Monitoring)**
Отслеживание выходных данных системы. Ключевой аспект — **смещение модели (model drift)**: модель под действием внешних факторов может деградировать. Необходимо:
- Фильтровать токсичность, иронию и недопустимые ответы
- Следить за соответствием поведения модели заданным инструкциям
- Внедрить систему обратной связи (feedback) — значительно снижает риск смещения

### Оповещения (Alerting)

Своевременно уведомлять администратора при обнаружении отклонений от нормы.

## Управление стоимостью (Cost Management)

Использование и размещение LLM-моделей может быть дорогостоящим, поэтому важно отслеживать и оптимизировать затраты.

### Структура затрат

| Тип размещения | Из чего складывается стоимость |
|----------------|-------------------------------|
| **Self-hosted** | Электричество + поддержка оборудования |
| **Облачный хостинг** | Время работы модели (время * тариф) |
| **API (внешний провайдер)** | Количество токенов (input + output) |

### Три способа оптимизации стоимости

**1. Выбор правильной модели**
Не обязательно везде использовать самую мощную и дорогую модель. Лучше использовать несколько небольших специализированных моделей, чем пытаться решить всё одной большой.

**2. Оптимизация промптов**
- Писать промпты максимально короко, но содержательно — каждое слово это токены
- Строго контролировать длину ответа: указывать количество слов, предложений, параграфов
- Чем короче промпт при сохранении смысла — тем дешевле

**3. Оптимизация количества вызовов**
- **Батчинг**: объединять группу вызовов в один запрос к модели
- **Кэширование**: при повторяющихся вопросах сохранять ответы и отдавать из кэша
- **Фильтрация задач**: всегда оценивать, нужно ли вообще вызывать LLM для данной задачи

> Для агентов батчинг вызовов особенно важен.

## Управление и безопасность (Governance & Security)

Пренебрежение этим аспектом влечёт серьёзные последствия.

- **Управление (Governance)** — политики, руководства и фреймворки разработки, внедрения и использования LLM-приложений
- **Безопасность (Security)** — меры по предотвращению несанкционированного доступа, утечек данных, враждебных атак и неправомерного использования

### Контроль доступа

Стандартный подход — **ролевой контроль доступа (RBAC)**:
- Разрешения назначаются ролям, пользователи назначаются ролям
- Все API должны соответствовать стандартам безопасности
- Рекомендуется модель безопасности **Zero Trust**: каждый пользователь проходит аутентификацию, авторизацию и постоянную проверку
- При работе с RAG важно, чтобы приложение принимало правильную роль для каждого запроса

### Основные угрозы

| Угроза | Описание | Меры защиты |
|--------|----------|-------------|
| **Prompt Injection** | Манипуляция полями ввода для выполнения несанкционированных команд | Использовать инструменты обнаружения атак; блокировать известные враждебные промпты; считать промпты потенциально раскрытыми |
| **Манипуляция выводом** | Изменение выходных данных LLM; вывод может использоваться в последующих атаках | Не предоставлять приложению лишних полномочий; цензурировать нежелательные выводы |
| **DoS-атаки** | Засыпание приложения запросами — проблемы с затратами, доступностью, производительностью | Ограничение частоты запросов (rate limiting); ограничение ресурсов на запрос |
| **Заражение данных (Data Poisoning)** | Введение ложных или вредоносных данных в обучающий набор | Получать данные из надёжных источников; использовать фильтры при обучении; цензура вывода |

> Актуальный список угроз для LLM-приложений: [OWASP Top 10 for LLMs](https://owasp.org/www-project-top-10-for-large-language-model-applications/)
